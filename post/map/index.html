Map结构

type hmap struct {

	count     int // 当前map中有多少个key,就是len()拿到的值
	flags     uint8
	B         uint8  //有2^B个桶
	noverflow uint16 // 使用的溢出桶数量
	hash0     uint32 // hash种子

	buckets    unsafe.Pointer //桶的地址
	oldbuckets unsafe.Pointer // 如果正在进行迁移，这是老桶的地址
	nevacuate  uintptr        //渐进式扩容的情况下的记录下一个要被迁移的旧桶的编号

	extra *mapextra // optional fields
}

概览图

map插入元素(简易流程)？

从上面的数据结构中我们可以看出来，map的桶的个数是2B次方个，那么桶的下表就是2B-1,这样的数字我们&上任何数字，那么这个值一定在0-2^B-1。那么我们保证数字的散列性，就能尽可能让key分散到各个桶中。

例如 现在有16桶那么例如我现在插入一个key进行hash运算后为101001101
101001101
&000001111
0000001101
经过结算，我们发现，这个key就会落在index为13的桶中

什么时候会扩容？

当负载因子大于6.5的时候就会发生扩容(hmap.count / 2^hmap.B > 6.5)

为什么是6.5?
因为经过测算，在负载因子等于6.5，浪费的空间，跟溢出桶的个数有一个平衡吧。

这种情况下会进行翻倍扩容，并且在进行数据迁移时进行渐进式迁移

当溢出桶过多的时候()

  • 如果常规桶数目不大于2^15，那么使用的溢出桶数目超过常规桶就算是多了
  • 如果常规桶数目大于215，那么使用溢出桶数目一旦超过215就算多了

其实就是很多数据被删除了，会导致这种情况的发生。进行等量扩容。这种情况下，如果把这些键值对重新安排到等量的新桶中，虽然哈希值没变，常规桶数目没变，每个键值对还是会选择与旧桶一样的新桶编号，但是能够存储的更加紧凑，进而减少溢出桶的使用

如何进行渐进式迁移？

例如本来原来有4个桶，现在扩容后是8个桶

如果你原来是在1号桶上,那么你的最后两位一定是0跟1。
xxxxxx01
00000011

现在新桶的个数是8。那么如果原来的hash值的第三位是0，就分配在新桶的1号桶上。是1就分配在5号桶上。
xxxxx0/101
00000 1 11
